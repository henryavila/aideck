## Summary
The plan has several internal contract breaks that will force redesign during implementation. The largest risks are an inconsistent MCP tool surface, a frontmatter mutation helper that cannot validate the entities it mutates without corrupting or losing markdown body handling, and late introduction of resolution semantics after the backend/schema/parser work is already complete.

Several smoke and DoD checks are also inconsistent with earlier implementation choices, so a team could complete the written steps and still fail the release checklist.

## Findings

**Evidence:**

Covers F4 (read half).

Frontmatter writer must preserve narrative body verbatim. Covers F4 (mutate half).

08. Covers F4 (remainder) + F13 (shell+manual) + data side of F11/F12.

**Claim:** The MCP tool count is not implementable as written because the listed surface is 7 read + 9 mutate + 8 remainder tools, while later contracts hardcode “18 tools”.

**Impact:** Implementers will either expose 24 tools and fail `toolCount: 18`/smoke checks, or trim tools and fail the step-level DoD for the omitted category.

**Recommendation:** Create one authoritative MCP tool index, set the actual count from that list, and update steps 06-08, `aideck_schema_version.toolCount`, and closeout smoke checks to the same number.

**Confidence:** high

### F-002 [critical] viability — docs/implementation/07-mcp-mutate.md:27-29

**Evidence:**
```md
1. `frontmatter.ts`:
   - `async function mutateFrontmatter<T>(path, parse: (raw) => Result<T, ErrorResponse>, mutate: (current: T) => T, serialize: (mutated: T) => unknown): Promise<Result<T, ErrorResponse>>`.
   - Lê arquivo, split via `splitFrontmatter`, `parse` (do Zod), `mutate`, `serialize` (objeto frontmatter), `serializeFrontmatter` (string final).
```

**Claim:** `mutateFrontmatter` parses only the YAML frontmatter with Zod, but earlier parser semantics require `Plan.narrative` and `Initiative.body` to be attached from the markdown body before validation.

**Impact:** Every mutation of a Plan or Initiative can fail validation for missing body fields; if a developer works around this by serializing the full validated object, `narrative`/`body` can be written into frontmatter and break byte-equal body preservation.

**Recommendation:** Make `mutateFrontmatter` entity-aware: split the file, parse YAML, attach `narrative` or `body` before validation/mutation, then strip that markdown field before serializing YAML while passing the original body unchanged.

**Confidence:** high

## Summary
The plan conflicts with hard project constraints and contains several internal contract breaks. The blocker is that multiple steps require writing canonical entity files under `plans/` and `initiatives/`, while the external constraint states aiDeck may only write to `annotations/`, `highlights/`, and `inbox/`.

Beyond that, the MCP surface is internally inconsistent, stdio-only MCP is treated as if it can be bundled into a normal `serve` daemon, late schema additions invalidate earlier backend steps, hash routing contradicts documented smoke URLs, and `not_implemented` is required despite the closed `ErrorResponse.code` enum excluding it.
