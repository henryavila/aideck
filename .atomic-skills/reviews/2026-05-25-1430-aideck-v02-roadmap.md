## Summary
The plan has implementability gaps around external file contracts, event retention, static report serving, and response schemas. Several endpoints/tools are specified at the name level but omit the payload shape or storage contract needed for independent implementation and consumer compatibility.

The highest-risk issue is F2: aiDeck is supposed to read state produced by an external consumer, but the plan does not define the files, glob, JSONL name, or version negotiation it will read. That makes the parser and endpoint impossible to implement consistently.

## Findings

### F-001 [critical] dependency break — .atomic-skills/_drafts/aideck-v02-roadmap-source.md:103-104

**Evidence:**
```md
- **T-001 — Schema `DispatchBatchState`.** Definir schema em `src/schemas/dispatch.ts` para o formato JSONL de progresso por agente (campos: agentId, batchId, status, taskId, progress, timestamp). Incluir `schemaVersion: '0.1'`. Dependencia externa: o diretorio `.atomic-skills/parallel-dispatch/<batch-id>/` e criado pelo consumer (`atomic-skills` skill `parallel-dispatch`), nao pelo aiDeck. aiDeck apenas le e projeta.
- **T-002 — Parser de batch state.** Parsear `.atomic-skills/parallel-dispatch/<batch-id>/` usando o schema de T-001. Projetar estado consolidado do batch (agentes ativos, progresso agregado, erros). O watcher deve ser estendido para observar `parallel-dispatch/` quando o diretorio existir (verified_by: src/server/watcher.ts observa .atomic-skills/**).
```

**Claim:** F2 depends on an external directory created by `atomic-skills`, but the plan never defines which file names, globs, manifest, or JSONL streams aiDeck must read inside that directory.

**Impact:** Two developers can implement incompatible producers and consumers: aiDeck may watch or parse the wrong files, `get_dispatch_status` can return empty data for valid batches, and the UI cannot rely on the endpoint for active dispatch state.

**Recommendation:** Define the external read contract explicitly in F2 before parser work: required files, optional files, glob patterns, record schema per file, batch discovery rules, and behavior for unknown `schemaVersion`.

**Confidence:** high

### F-003 [major] viability — .atomic-skills/_drafts/aideck-v02-roadmap-source.md:67

**Evidence:**
```md
- **T-001 — Event log persistente.** Cada evento SSE tambem grava num append-only JSONL em `.atomic-skills/status/events.jsonl` (por projeto). Maximo 10k linhas; rotacao para `events-<YYYY-MM>.jsonl` (sem compressao — arquivos permanecem legíveis e buscaveis). Rotacao via rename atomico; leituras nao bloqueiam escrita.
```

**Claim:** The rotation target `events-<YYYY-MM>.jsonl` is not unique for multiple 10k-line rotations in the same month.

**Impact:** A busy project that exceeds 20k events in one month can collide with an existing rotated file; an implementation that renames atomically to the same path will either fail rotation or overwrite prior events, breaking the “sem perda de eventos” gate.

**Recommendation:** Specify a collision-free rotation naming scheme, such as `events-<YYYY-MM>-<sequence>.jsonl` or timestamped names, and require tests for two rotations within the same month.

**Confidence:** high
