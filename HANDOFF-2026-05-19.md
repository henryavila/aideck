## Why this change

The atomic-skills session was crystallizing how skills would integrate with aiDeck via MCP. Question came up: "how does a skill detect that aiDeck is actually running?"

Reviewing aiDeck contracts: no spec existed. Port `127.0.0.1:7777` was mentioned, MCP config was mentioned, but **the consumer-side detection mechanism was undefined**. That's a real gap — a downstream consumer would invent its own (potentially incompatible) detection convention.

The chosen design (file + MCP availability) is graceful-degradation friendly: skills don't need aiDeck installed to work; aiDeck is an optimization layer.

### Step 09 (CLI/demo) — what to add

Currently your step covers `aideck serve | demo | mcp` with SIGINT cleanup. Add to the same step (or split into 09b if you prefer):

**New requirements**:

1. On `aideck serve` startup (and `aideck mcp` if it serves HTTP):
   - Ensure `~/.aideck/` directory exists (mode 0700)
   - Write `~/.aideck/runtime.json` (mode 0600) with the documented shape
   - Include `process.pid` from `process.pid`
   - Include `port`, `url`, `modes`, `startedAt`, `version` from package.json + config

2. On graceful shutdown (SIGINT, SIGTERM, normal exit):
   - Remove `~/.aideck/runtime.json`
   - This is part of the same cleanup flow as your existing demo tmpdir cleanup

3. On startup, check if `~/.aideck/runtime.json` already exists:
   - If yes, read PID and try `process.kill(pid, 0)` (POSIX liveness check)
   - If process is alive: warn or refuse to start (port collision would block anyway, but explicit message is better)
   - If process is dead: stale file — overwrite

**New tests** (add to step 09 DoD):
- [ ] `aideck serve` creates `~/.aideck/runtime.json` with valid JSON
- [ ] PID matches `process.pid`
- [ ] SIGINT removes the file
- [ ] Kill -9 leaves stale file; subsequent start detects + overwrites
- [ ] File mode is 0600; directory mode is 0700
