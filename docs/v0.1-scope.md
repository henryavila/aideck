# v0.1 Scope

## Goal

Working aiDeck **backend runtime** (HTTP + SSE + MCP + watcher) sufficient to drive an external dashboard UI. Sufficient to dogfood inside atomic-skills development. The dashboard UI is built and shipped by the consumer (e.g. the `atomic-skills` repo) and served by aiDeck via `--static-dir`.

## In-scope

### Runtime
- Hono HTTP server on `127.0.0.1:7777` (configurable port)
- chokidar watcher on `.atomic-skills/**`
- SSE endpoint pushing canonical-data change events to browser
- REST API: `GET /api/state/:consumer/:slug?`, `GET /api/help`
- Optional static SPA serving via `aideck serve --static-dir <path>` (catch-all → `index.html`, with API/SSE passthrough)

### MCP server
- 18 tools across 5 categories: read, mutate, exit-gate, feedback, meta (see [`mcp-tools.md`](./mcp-tools.md))
- stdio transport (Claude Code, Cursor compatibility)
- Runs alongside HTTP server (same process)
- Tools cover full operational loop: read state → mark done → verify gates → annotate → inbox

### CLI
- `aideck serve [--port=N] [--no-mcp]`
- `aideck demo`
- `aideck mcp` (MCP-only)
- `aideck --help`

### Detection / lifecycle
- Writes `~/.aideck/runtime.json` on startup (PID + URL + port + version)
- Removes the file on graceful shutdown; PID liveness check for stale recovery
- MCP tool availability is the canonical detection signal for AI agents
- See [`integration-spec.md`](./integration-spec.md) for full spec

### Demo fixtures
- 1 sample plan (3 phases, 12 tasks, mixed states)
- 1 sample initiative (standalone, no parent plan)
- 1 ad-hoc session entry
- 3 sample annotations, 2 highlights

## Out-of-scope for v0.1

- **Dashboard UI** → lives in the `atomic-skills` sibling repo (`atomic-skills/src/dashboard/`), shipped as a built SPA bundle and consumed by aiDeck via `--static-dir`. aiDeck v0.1 does not bundle a frontend.
- parallel-dispatch / parallel-dispatch-audit renderers → v0.2 (UI-side, in `atomic-skills`)
- hunt / review renderers → v0.3 (UI-side, in `atomic-skills`)
- Cross-cutting views (home, timeline, repo health) → v0.4 (UI-side, in `atomic-skills`)
- Custom consumer registration (third-party tools) → v0.2+
- Dashboard-side mutation (click to mark task done) → v0.2+ (UI-side)
- Authentication / multi-user → never (local-first only)
- Cloud sync → never (local-first only)

1. project-status redesigned to plan/phase/task hierarchy (per design discussion 2026-05-19)
